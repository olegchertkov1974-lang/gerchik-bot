# CLAUDE.md — память проекта Gerchik Trading Bot

Поведенческие правила для Claude Code. Объединяют принципы Карпати, контекст проекта и инфраструктуру.

---

## 1. Проект

- **Сервер:** root@193.242.109.236 (Beget VPS)
- **ОС:** Ubuntu 22.04
- **Ресурсы:** 2 GB RAM, 2 GB swap, 30 GB NVMe
- **Node.js:** v20.20.0
- **PM2 процессы:** `gerchik-bot`, `gerchik-dashboard` (порт 3002)
- **Папка проекта:** /opt/trading-bot/trading-bot
- **Стратегия:** /opt/trading-bot/var/strategies/gerchik-levels.js (вне папки проекта)
- **База сделок:** data/trades.db (SQLite)
- **Логи:** logs/trading-bot.log
- **Биржа:** Bybit **demo/testnet** (api-demo.bybit.com) — на реальные деньги пока НЕ переведён

### Методология Герчика
- Лимитные ордера с PostOnly
- Мульти-таймфрейм: 1D для уровней, 4H для трендового фильтра, 5m для входа
- Минимум 1:3 R:R
- Stop Market для SL
- Активные фильтры: запас хода, запил, глубокие ЛП, энергия 5m, detectMarketRegime (ATR14 + MA5/MA20), entry filter по режиму, динамический риск (3 стопа → 50%, 5 стопов → пауза)

### Конфигурация стратегии
- Три тейк-профита: TP1 30% на 1R, TP2 40% на 2R, TP3 30% на 3R
- Стоп в безубыток после TP1
- AI фильтр: `google/gemini-2.5-flash` через OpenRouter

### Основные команды
```bash
pm2 status                        # статус
pm2 logs gerchik-bot --lines 50   # логи
pm2 restart gerchik-bot           # рестарт
pm2 stop gerchik-bot              # остановить
free -h                           # память
```

## 2. Правила поведения

### 2.1. Сначала думай, потом пиши код

**Не угадывай. Не скрывай неопределённость. Называй развилки.**

Перед реализацией:
- Явно озвучь свои допущения. Если не уверен — спроси.
- Если задачу можно решить несколькими способами — перечисли их, не выбирай молча.
- Если есть более простой подход — скажи об этом.
- Если что-то непонятно — остановись, назови что именно, спроси.

### 2.2. Простота прежде всего

**Минимум кода, который решает задачу. Ничего лишнего.**

- Никаких фич сверх того, что просили.
- Никаких абстракций для одноразового кода.
- Никакой «гибкости» и «настраиваемости», которую не просили.
- Никакой обработки ошибок для невозможных сценариев.
- Написал 200 строк, а можно 50 — перепиши.

Критерий: «Сказал бы опытный инженер, что это избыточно?» Если да — упрощай.

### 2.3. Хирургические изменения

**Трогай только то, что необходимо. Убирай только своё.**

При редактировании существующего кода:
- Не «улучшай» соседний код, комментарии или форматирование.
- Не рефакторь то, что не сломано.
- Придерживайся текущего стиля, даже если ты бы сделал иначе.
- Заметил мёртвый код не по теме — упомяни, но не удаляй.

После своих изменений:
- Удали импорты / переменные / функции, которые стали лишними **из-за твоих правок**.
- Не трогай уже существующий мёртвый код без явной просьбы.

Тест: каждая изменённая строка должна прямо следовать из запроса.

### 2.4. Выполнение через цели

**Определи критерий успеха. Крутись в цикле до проверки.**

Переформулируй задачи в проверяемые цели:
- «Добавь валидацию» → «Напиши тесты на невалидный ввод, затем сделай их зелёными»
- «Почини баг» → «Напиши тест, воспроизводящий баг, затем сделай его зелёным»

Для многошаговых задач — кратко озвучь план:
```
1. [Шаг] → проверка: [что проверяем]
2. [Шаг] → проверка: [что проверяем]
3. [Шаг] → проверка: [что проверяем]
```

## 3. Безопасность (secrets-guard)

- **Никогда** не читать, не копировать и не передавать реальные API-ключи из .env
- Если нужен пример — использовать placeholder: `YOUR_API_KEY_HERE`, `YOUR_SECRET_HERE`
- В коде и логах — только плейсхолдеры: `BYBIT_API_KEY`, `OPENROUTER_API_KEY` и т.д.
- **НЕ коммить** в git: .env, data/trades.db, logs/*, node_modules/, reports/
- Изменения .env делать через редактирование файла, НЕ через echo с реальными ключами в команде
- Перед любым git commit — напомни проверить, что секреты не попали в diff

## 4. Работа с ботом (протокол изменений)

При запросе на изменение кода:

1. **СНАЧАЛА** опиши на русском языке простыми словами: что изменится, в каком файле, какие побочные эффекты
2. **ЖДИ явного подтверждения** от Олега ("да", "подтверждаю", "го") перед тем как писать код
3. **ПОСЛЕ подтверждения** — вноси изменения
4. **ОБЯЗАТЕЛЬНО в конце** выполняй рестарт и проверку:
```bash
pm2 restart gerchik-bot && sleep 3 && pm2 logs gerchik-bot --lines 20
```
5. Если рестарт показывает ошибки — сразу сообщай, предлагай откат
